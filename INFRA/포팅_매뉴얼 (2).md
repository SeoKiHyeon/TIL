## 1. 개요

본 문서는 ARnnect 프로젝트의 인프라 환경을 새로운 서버에 포팅하기 위한 매뉴얼입니다.

### 기술 스택

| 구분 | 기술 | 버전 |
|------|------|------|
| 컨테이너 | Docker & Docker Compose | 최신 |
| 리버스 프록시 | Nginx | 1.29.4-alpine |
| SSL 인증서 | Certbot (Let's Encrypt) | v5.2.2 |
| 데이터베이스 | MySQL | 8.0 |
| 캐시 | Redis | 7.4 |
| CI/CD | Jenkins | LTS |
| 백엔드 | Spring Boot | - |
| 프론트엔드 | React (Nginx 서빙) | - |
| AI 추천 서비스 | BentoML (RunPod GPU) | - |

### 환경 구분

| 환경 | 용도 | HTTPS 포트 | HTTP 포트 |
|------|------|-----------|----------|
| Release (운영) | 실서비스 | 443 | 80 |
| Develop (개발) | 개발/테스트 | 8001 | 8000 |
| Jenkins | CI/CD 관리 | 9000 | - |

```
                   |  :9000 (jenkins)     |

                              |
              ┌───────────────┼───────────────┐
              |               |               |
     ┌────────┴──────┐ ┌─────┴─────┐  ┌──────┴──────┐
     | Release 환경   | | Develop   |  |   Jenkins   |
     | prod-fe(:80)   | | test-fe   |  |   (:8080)   |
     | prod-be(:8080) | | test-be   |  └─────────────┘
     | prod-redis     | | test-redis|
     └────────┬──────┘ └─────┬─────┘
              |               |
              └───────┬───────┘
                      |
              ┌───────┴───────┐
              |    MySQL      |

              |  공용 DB       |
```

### Docker 네트워크 구성

```
gateway-net (bridge)  ←─ 외부 트래픽 라우팅용
  ├── gateway-nginx
  ├── jenkins
  ├── prod-fe, prod-be
  └── test-fe, test-be

infra-net (bridge)    ←─ DB 접근용 내부 네트워크
  ├── mysql
  ├── prod-be
  └── test-be

default (bridge)      ←─ Redis 접근용 로컬 네트워크
  ├── prod-be ↔ prod-redis
  └── test-be ↔ test-redis
```

## 4. 서버 초기 설정

### 4.1 Docker 설치

```bash
# 패키지 업데이트
sudo apt-get update
sudo apt-get upgrade -y

# Docker 설치를 위한 필수 패키지
sudo apt-get install -y \
    ca-certificates \
    curl \
    gnupg \
    lsb-release

# Docker 공식 GPG 키 추가
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
    sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

# Docker 저장소 추가
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Docker 설치
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin

# Docker 서비스 시작 및 자동 시작 설정
sudo systemctl start docker
sudo systemctl enable docker

# 현재 사용자를 docker 그룹에 추가 (sudo 없이 docker 사용)
sudo usermod -aG docker $USER
newgrp docker
```

### 4.2 Docker Compose 설치 확인

```bash
# Docker Compose v2 확인 (docker-compose-plugin 포함)
docker compose version
```

### 4.3 타임존 설정

```bash
sudo timedatectl set-timezone Asia/Seoul
```

### 4.4 방화벽 설정 (UFW)

```bash
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw allow 8000/tcp
sudo ufw allow 8001/tcp
sudo ufw allow 8806/tcp

```

## 6. Gateway 설정 (Nginx + SSL)

### 6.1 docker-compose.yml

**파일 위치:** `~/Infra/gateway/docker-compose.yml`

```yaml
services:
  nginx:
    image: nginx:1.29.4-alpine
    container_name: gateway-nginx
    restart: always
    ports:
      - "80:80"     # prod HTTP
      - "443:443"   # prod HTTPS
      - "8000:8000" # dev HTTP
      - "8001:8001" # dev HTTPS
      - "9000:9000" # Jenkins
    volumes:
      - ./nginx/conf.d:/etc/nginx/conf.d:ro               # 라우팅 규칙
      - ./nginx/logs:/var/log/nginx                       # 로그 저장
      # 인증서와 검증 파일 공유
      - ./certbot/conf:/etc/letsencrypt
      - ./certbot/www:/var/www/certbot
    networks:
      - gateway-net

  certbot:
    image: certbot/certbot:v5.2.2
    container_name: gateway-certbot
    volumes:
      - ./certbot/conf:/etc/letsencrypt
      - ./certbot/www:/var/www/certbot

# 여기서 네트워크를 생성
# 다른(Release, Develop) docker-compose 파일들은 이 네트워크를 'external'로 참조
networks:
  gateway-net:
    name: gateway-net
    driver: bridge
```

### 6.2 Nginx 설정 (default.conf)

**파일 위치:** `~/Infra/gateway/nginx/conf.d/default.conf`

> **중요:** 아래 설정에서 `i14e107.p.ssafy.io` 부분을 실제 도메인으로 변경해야 합니다.

```nginx
# ============================================
# Release (운영) 환경
# ============================================

# HTTP → HTTPS 리다이렉트
server {
    listen 80;
    server_name i14e107.p.ssafy.io;
    server_tokens off;

    # Let's Encrypt 인증서 갱신용
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    # 모든 HTTP 요청을 HTTPS로 리다이렉트
    location / {
        return 301 https://$host$request_uri;
    }
}

# HTTPS (운영)
server {
    listen 443 ssl;
    server_name i14e107.p.ssafy.io;
    server_tokens off;

    # [인증서 경로]
    ssl_certificate /etc/letsencrypt/live/i14e107.p.ssafy.io/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/i14e107.p.ssafy.io/privkey.pem;

    # [최적화 옵션]
    client_max_body_size 20M;
    gzip on;
    gzip_types text/plain application/json text/css application/javascript;

    # [프론트엔드] React SPA
    location / {
        proxy_pass http://prod-fe:80;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # [백엔드] Spring Boot API
    location /api {
        proxy_pass http://prod-be:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # [3D 모델 파일] 1년 캐싱
    location ~* \.(glb|gltf|obj|fbx)$ {
        proxy_pass http://prod-fe:80;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-Proto $scheme;

        expires 1y;
        add_header Cache-Control "public, no-transform";
        access_log off;
    }
}

# ============================================
# Develop (개발) 환경
# ============================================

# HTTP → HTTPS 리다이렉트 (개발)
server {
    listen 8000;
    server_name i14e107.p.ssafy.io;
    server_tokens off;

    location / {
        return 301 https://$host:8001$request_uri;
    }
}

# HTTPS (개발)
server {
    listen 8001 ssl;
    server_name i14e107.p.ssafy.io;
    server_tokens off;

    ssl_certificate /etc/letsencrypt/live/i14e107.p.ssafy.io/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/i14e107.p.ssafy.io/privkey.pem;

    client_max_body_size 20M;
    gzip on;
    gzip_types text/plain application/json text/css application/javascript;

    # [프론트엔드] React SPA
    location / {
        proxy_pass http://test-fe:80;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # [백엔드] Spring Boot API
    location /api {
        proxy_pass http://test-be:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # [Swagger] API 문서 - OpenAPI v3 스펙
    location /v3 {
        proxy_pass http://test-be:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # [Swagger] UI 페이지
    location /swagger-ui {
        proxy_pass http://test-be:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # [3D 모델 파일] 1년 캐싱
    location ~* \.(glb|gltf|obj|fbx)$ {
        proxy_pass http://test-fe:80;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-Proto $scheme;

        expires 1y;
        add_header Cache-Control "public, no-transform";
        access_log off;
    }
}

# ============================================
# Jenkins CI/CD
# ============================================
server {
    listen 9000 ssl;
    server_name i14e107.p.ssafy.io;

    ssl_certificate /etc/letsencrypt/live/i14e107.p.ssafy.io/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/i14e107.p.ssafy.io/privkey.pem;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;

    location / {
        proxy_pass http://jenkins:8080;

        proxy_set_header Host $host:$server_port;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 젠킨스 경고 해결 (http → https 리다이렉트 인식)
        proxy_redirect http:// https://;
    }
}
```

## 7. 공용 서비스 설정 (MySQL + Jenkins)

### 7.1 환경변수 파일 생성

**파일 위치:** `~/Infra/common/.env`

```env
MYSQL_ROOT_PASSWORD=<루트_비밀번호>
MYSQL_USER=<사용자명>
MYSQL_PASSWORD=<사용자_비밀번호>
MYSQL_DATABASE=<데이터베이스명>
```

> **주의:** `.env` 파일은 절대 Git에 커밋하지 마세요. `.gitignore`에 추가하세요.

### 7.2 docker-compose.yml

**파일 위치:** `~/Infra/common/docker-compose.yml`

```yaml
version: '3.8'

services:
  # 1. MySQL (공용 데이터베이스)
  mysql:
    image: mysql:8.0
    container_name: mysql
    restart: always
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
      MYSQL_USER: ${MYSQL_USER}
      MYSQL_PASSWORD: ${MYSQL_PASSWORD}
      MYSQL_DATABASE: ${MYSQL_DATABASE}
      TZ: Asia/Seoul
    volumes:
      - ./mysql:/var/lib/mysql              # 데이터 영구 저장
    ports:
      - "8806:3306"
    command:
      - --character-set-server=utf8mb4
      - --collation-server=utf8mb4_unicode_ci
    networks:
      - infra-net
    deploy:
      resources:
        limits:
          memory: 2G

  # 2. Jenkins (CI/CD 파이프라인)
  jenkins:
    image: jenkins/jenkins:lts
    container_name: jenkins
    restart: always
    user: root  # Docker 소켓 접근 권한을 위해 루트로 실행
    volumes:
      - ./jenkins:/var/jenkins_home
      - /var/run/docker.sock:/var/run/docker.sock  # 호스트 Docker 조종
      - /usr/bin/docker:/usr/bin/docker             # Docker CLI 사용
    environment:
      - TZ=Asia/Seoul
    networks:
      - gateway-net
    deploy:
      resources:
        limits:
          memory: 2G

# [네트워크 선언]
networks:
  gateway-net:
    external: true
  infra-net:
    name: infra-net
    driver: bridge
```

### 7.3 핵심 설정 설명

| 항목 | 설명 |
|------|------|
| MySQL 포트 `8806:3306` | 호스트의 8806 포트로 MySQL 접속 가능. 외부 DB 클라이언트 사용 시 활용 |
| MySQL `utf8mb4` | 한글 및 이모지 지원을 위한 문자셋 설정 |
| Jenkins `user: root` | Docker 소켓(`/var/run/docker.sock`)에 접근하기 위해 필수 |
| Jenkins Docker 마운트 | Jenkins 컨테이너 내부에서 호스트 Docker 명령어를 실행 (DinD 대신 DooD 방식) |
| `gateway-net: external` | Gateway에서 생성한 네트워크를 참조. **Gateway를 먼저 기동해야 함** |

## 11. SSL 인증서 발급 (Let's Encrypt)

### 11.1 초기 인증서 발급 (최초 1회)

SSL 인증서가 없는 상태에서 Nginx를 시작하면 에러가 발생합니다. 다음 절차로 해결합니다.

#### 방법 A: 임시 Nginx 설정으로 인증서 발급

**1단계:** SSL이 없는 임시 Nginx 설정 생성

```bash
# 임시 설정 백업
cp ~/Infra/gateway/nginx/conf.d/default.conf ~/Infra/gateway/nginx/conf.d/default.conf.ssl-backup

# 임시 HTTP-only 설정으로 변경
cat > ~/Infra/gateway/nginx/conf.d/default.conf << 'EOF'
server {
    listen 80;
    server_name <도메인>;
    server_tokens off;

    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    location / {
        return 200 'SSL setup in progress';
        add_header Content-Type text/plain;
    }
}
EOF
```

**2단계:** Gateway 재시작 및 인증서 발급

```bash
cd ~/Infra/gateway

# Nginx 재시작
docker compose restart nginx

# Certbot으로 인증서 발급
docker compose run --rm certbot certonly \
    --webroot \
    --webroot-path=/var/www/certbot \
    -d <도메인> \
    --email <이메일> \
    --agree-tos \
    --no-eff-email
```

**3단계:** 원래 설정 복구 및 재시작

```bash
# 원래 Nginx 설정 복구
cp ~/Infra/gateway/nginx/conf.d/default.conf.ssl-backup \
   ~/Infra/gateway/nginx/conf.d/default.conf

# Nginx 재시작
docker compose restart nginx
```

### 11.2 인증서 자동 갱신

Let's Encrypt 인증서는 90일마다 갱신이 필요합니다. crontab에 자동 갱신을 등록합니다.

```bash
# crontab 편집
crontab -e

# 매달 1일 새벽 3시에 인증서 갱신 시도
0 3 1 * * cd ~/Infra/gateway && docker compose run --rm certbot renew && docker compose exec nginx nginx -s reload
```

#### 방법 B: Jenkins 웹 UI에서 Pipeline 직접 입력

Job 설정 페이지에서 **Pipeline** 섹션의 **Definition**을 `Pipeline script`로 선택하고, 아래 파이프라인 스크립트를 입력합니다.

**develop Job Pipeline:**

```groovy
pipeline {
    agent any

    environment {
        DOCKER_ID   = 'csw197'
        REPO_NAME   = 'sf'
        GIT_URL     = 'https://lab.ssafy.com/s14-webmobile2-sub1/S14P11E107'
        CRED_GIT    = 'gitlab-login'
        BRANCH_NAME = 'develop'

        SERVER_IP   = 'i14e107.p.ssafy.io'
        SERVER_USER = 'ubuntu'
        CRED_SSH    = 'server-ssh-key'
        DEPLOY_DIR  = '/home/ubuntu/infra/develop'

        TAG_BE = 'arnnect-be-dev'
        TAG_FE = 'arnnect-fe-dev'

        MATTERMOST_URL = credentials('mattermost-webhook-url')
    }

    tools {
        nodejs 'node-24'
        jdk 'jdk-17'
    }

    parameters {
        booleanParam(name: 'FORCE_BUILD_BE', defaultValue: false, description: '체크 시 Backend 강제 빌드')
        booleanParam(name: 'FORCE_BUILD_FE', defaultValue: false, description: '체크 시 Frontend 강제 빌드')
    }

    stages {
        stage('Clone Repository') {
            steps {
                checkout([
                    $class: 'GitSCM',
                    branches: [[name: "*/${BRANCH_NAME}"]],
                    doGenerateSubmoduleConfigurations: false,
                    extensions: [],
                    submoduleCfg: [],
                    userRemoteConfigs: [[credentialsId: "${CRED_GIT}", url: "${GIT_URL}"]]
                ])
            }
        }

        stage('Smart Build & Transfer') {
            parallel {
                stage('Backend') {
                    when {
                        anyOf {
                            changeset "BE/**"
                            expression { return params.FORCE_BUILD_BE }
                        }
                    }
                    steps {
                        sshagent(credentials: ["${CRED_SSH}"]) {
                            script {
                                echo 'Backend 빌드 및 전송 시작...'
                                dir('BE') {
                                    sh 'chmod +x gradlew'
                                    sh './gradlew clean bootJar'
                                    sh "docker build -t ${DOCKER_ID}/${REPO_NAME}:${TAG_BE} ."
                                    sh "docker save ${DOCKER_ID}/${REPO_NAME}:${TAG_BE} | gzip > ${TAG_BE}.tar.gz"
                                    sh "scp -o StrictHostKeyChecking=no ${TAG_BE}.tar.gz ${SERVER_USER}@${SERVER_IP}:${DEPLOY_DIR}/${TAG_BE}.tar.gz"
                                    sh "rm ${TAG_BE}.tar.gz"
                                }
                            }
                        }
                    }
                }

                stage('Frontend') {
                    when {
                        anyOf {
                            changeset "FE/**"
                            expression { return params.FORCE_BUILD_FE }
                        }
                    }
                    steps {
                        sshagent(credentials: ["${CRED_SSH}"]) {
                            script {
                                echo 'Frontend 빌드 및 전송 시작...'
                                dir('FE') {
                                    sh 'npm install'
                                    sh 'npm run build'
                                    sh "docker build -t ${DOCKER_ID}/${REPO_NAME}:${TAG_FE} ."
                                    sh "docker save ${DOCKER_ID}/${REPO_NAME}:${TAG_FE} | gzip > ${TAG_FE}.tar.gz"
                                    sh "scp -o StrictHostKeyChecking=no ${TAG_FE}.tar.gz ${SERVER_USER}@${SERVER_IP}:${DEPLOY_DIR}/${TAG_FE}.tar.gz"
                                    sh "rm ${TAG_FE}.tar.gz"
                                }
                            }
                        }
                    }
                }
            }
        }

        stage('Deploy to Server') {
            steps {
                sshagent(credentials: ["${CRED_SSH}"]) {
                    sh """
                        ssh -o StrictHostKeyChecking=no ${SERVER_USER}@${SERVER_IP} '
                            cd ${DEPLOY_DIR}
                            if [ -f "${TAG_BE}.tar.gz" ]; then
                                echo "Loading Backend Image..."
                                docker load < ${TAG_BE}.tar.gz
                                rm ${TAG_BE}.tar.gz
                            fi
                            if [ -f "${TAG_FE}.tar.gz" ]; then
                                echo "Loading Frontend Image..."
                                docker load < ${TAG_FE}.tar.gz
                                rm ${TAG_FE}.tar.gz
                            fi
                            docker compose up -d
                            docker image prune -f
                        '
                    """
                }
            }
        }
    }

    post {
        always {
            sh "docker image prune -f"
        }
        success {
            script {
                def author = sh(script: "git show -s --pretty=%an", returnStdout: true).trim()
                def message = "TEST 배포 성공 (Internal)\n" +
                              "- 프로젝트: ${env.JOB_NAME}\n" +
                              "- 빌드 번호: #${env.BUILD_NUMBER}\n" +
                              "- 작성자: ${author}\n" +
                              "- 결과: [Jenkins에서 보기](${env.BUILD_URL})"
                def payload = "{\"text\": \"${message}\"}"
                sh """
                    curl -i -X POST -H 'Content-Type: application/json' \
                    -d '${payload}' \
                    ${MATTERMOST_URL}
                """
            }
        }
        failure {
            script {
                def author = sh(script: "git show -s --pretty=%an", returnStdout: true).trim()
                def message = "TEST 배포 실패\n" +
                              "- 프로젝트: ${env.JOB_NAME}\n" +
                              "- 빌드 번호: #${env.BUILD_NUMBER}\n" +
                              "- 작성자: ${author}\n" +
                              "- 결과: [로그 확인하기](${env.BUILD_URL}console)"
                def payload = "{\"text\": \"${message}\"}"
                sh """
                    curl -i -X POST -H 'Content-Type: application/json' \
                    -d '${payload}' \
                    ${MATTERMOST_URL}
                """
            }
        }
    }
}
```

**release Job Pipeline:**

develop와 동일한 구조이며, 다음 환경변수만 다릅니다:

| 환경변수 | develop | release |
|---------|---------|---------|
| `BRANCH_NAME` | `develop` | `master` |
| `DEPLOY_DIR` | `/home/ubuntu/infra/develop` | `/home/ubuntu/infra/release` |
| `TAG_BE` | `arnnect-be-dev` | `arnnect-be-release` |

### 12.7 GitLab Webhook 설정

GitLab의 Webhook을 이용하여 특정 브랜치에 Push 이벤트 발생 시 Jenkins 빌드가 자동으로 트리거되도록 설정합니다.

#### Jenkins 측 설정 (Build Trigger)

각 Job의 설정 페이지에서 **Build Triggers** 섹션을 구성합니다.

1. Job 설정 → **Build Triggers** → **Build when a change is pushed to GitLab** 체크
2. 아래 항목을 설정합니다:

**develop Job:**

| 항목 | 설정값 |
|------|--------|
| Push Events | 체크 |
| Approved Merge Requests | 체크 |
| Comment (note) events | 체크 (리트라이: `Jenkins please retry a build`) |
| Filter branches by | **All branches** |
| Secret Token | `Generate` 버튼 클릭하여 생성 → **복사해두기** |

**release Job:**

| 항목 | 설정값 |
|------|--------|
| Push Events | 체크 |
| Approved Merge Requests | 체크 |
| Comment (note) events | 체크 (리트라이: `Jenkins please retry a build`) |
| Filter branches by | **Filter branches by regex** → Target Branch Regex: `master` |
| Secret Token | `Generate` 버튼 클릭하여 생성 → **복사해두기** |

> Secret Token은 GitLab Webhook 설정 시 필요하므로 반드시 복사해둡니다.

#### GitLab 측 설정 (Webhook 등록)

GitLab 프로젝트에서 Webhook을 등록합니다. **develop**과 **release** 각각 하나씩, 총 2개의 Webhook을 등록해야 합니다.

**설정 경로:** GitLab 프로젝트 → `Settings` → `Webhooks` → `Add new webhook`

**Webhook 1: develop (개발 배포)**

| 항목 | 설정값 |
|------|--------|
| URL | `https://<도메인>:9000/project/develop` |
| Secret Token | Jenkins develop Job에서 생성한 Secret Token 입력 |
| Trigger | **Push events** 체크 → **Wildcard pattern** 선택 → `develop` 입력 |

**Webhook 2: release (운영 배포)**

| 항목 | 설정값 |
|------|--------|
| URL | `https://<도메인>:9000/project/release` |
| Secret Token | Jenkins release Job에서 생성한 Secret Token 입력 |
| Trigger | **Push events** 체크 → **Wildcard pattern** 선택 → `master` 입력 |

#### Webhook 테스트

GitLab Webhook 설정 하단의 `Test` → `Push events` 버튼을 클릭하여 Jenkins 빌드가 트리거되는지 확인합니다.

- 정상 동작 시: HTTP 200 응답
- Jenkins Job이 자동으로 빌드 시작

### 12.8 Mattermost Webhook 설정

빌드 결과(성공/실패)를 Mattermost 채널로 자동 알림하기 위한 설정입니다.

#### Mattermost 측 설정 (Incoming Webhook 생성)

1. Mattermost 접속 → 알림을 받을 채널 선택
2. 채널 헤더의 드롭다운 메뉴 → `Integrations` (또는 `통합`) 클릭
3. `Incoming Webhooks` → `Add Incoming Webhook` 클릭
4. 설정:
   - **Display Name**: `Jenkins CI/CD` (표시 이름)
   - **Description**: `Jenkins 빌드 알림` (설명)
   - **Channel**: 알림을 받을 채널 선택
5. `Save` 클릭 → 생성된 **Webhook URL**을 복사

> Webhook URL 형식: `https://<mattermost-domain>/hooks/<webhook-id>`

#### Jenkins 측 설정 (Webhook URL 등록)

Mattermost Webhook URL을 Jenkins Credentials에 등록합니다.

1. `Jenkins 관리` → `Credentials` → `(global)` → `Add Credentials`
2. 설정:
   - **Kind**: Secret text
   - **Secret**: Mattermost에서 복사한 Webhook URL 입력
   - **ID**: `mattermost-webhook-url`
   - **Description**: `Mattermost Webhook URL`
3. `Create` 클릭

> Pipeline 스크립트에서 `credentials('mattermost-webhook-url')`로 참조하여 빌드 성공/실패 시 자동으로 Mattermost 채널에 알림을 전송합니다.

|------|-----------|
| 성공 | 프로젝트명, 빌드 번호, 작성자, Jenkins 링크 포함 |
| 실패 | 프로젝트명, 빌드 번호, 작성자, 콘솔 로그 링크 포함 |

## 17. 문제 해결 (Troubleshooting)

### Q1: `network gateway-net declared as external, but could not be found`

**원인:** Gateway를 먼저 기동하지 않았습니다.

**해결:**
```bash
cd ~/Infra/gateway && docker compose up -d
```

### Q2: Nginx가 시작되지 않음 (SSL 인증서 관련)

**원인:** SSL 인증서가 아직 발급되지 않았습니다.

**해결:** [11. SSL 인증서 발급](#11-ssl-인증서-발급-lets-encrypt) 섹션을 참고하여 인증서를 먼저 발급하세요.

### Q3: Jenkins에서 Docker 명령어가 실행되지 않음

**원인:** Docker 소켓 마운트 또는 Docker CLI 경로 문제

**해결:**
```bash
# Docker 소켓 권한 확인
ls -la /var/run/docker.sock

# 필요 시 권한 부여
sudo chmod 666 /var/run/docker.sock

# Docker CLI 경로 확인
which docker  # /usr/bin/docker 이어야 함
```

### Q4: MySQL 접속 실패

**원인:** MySQL 초기화가 완료되지 않았거나, 네트워크 설정 오류

**해결:**
```bash
# MySQL 초기화 상태 확인
docker logs mysql | tail -20

# infra-net 네트워크에 백엔드가 연결되어 있는지 확인
docker network inspect infra-net
```

### Q5: 컨테이너가 계속 재시작됨

```bash
# 컨테이너 로그 확인
docker logs <컨테이너명>

# 메모리 사용량 확인
docker stats --no-stream
```

### Q6: 3D 모델 파일이 로드되지 않음

**원인:** `/home/ubuntu/src` 디렉토리에 파일이 없거나 권한 문제

**해결:**
```bash
# 디렉토리 존재 확인
ls -la /home/ubuntu/src

# 권한 부여
chmod -R 755 /home/ubuntu/src
```

### Q7: 프론트엔드 페이지 새로고침 시 404 에러

**원인:** Nginx `try_files` 설정 누락

**해결:** 프론트엔드 Nginx 설정(`develop/nginx-conf/default.conf` 또는 `release/nginx-conf/default.conf`)에 `try_files $uri $uri/ /index.html;` 이 포함되어 있는지 확인하세요.

## 18. AI 추천 서비스 - RunPod 배포

BentoML 기반 AI 추천 서비스(Reco Service)를 RunPod GPU Pod에 Docker 컨테이너로 배포하는 방법을 설명합니다.

### 18.1 서비스 스펙

| 항목 | 값 |
|------|------|
| 프레임워크 | BentoML |
| 엔트리포인트 | `service:RecoService` |
| 기본 포트 | `8000` |
| GPU 요구사항 | CUDA 지원 GPU 권장 |

핵심 구성요소:
- SASRec, Two-Tower 모델 체크포인트
- 매핑 파일 (작품 ↔ 인덱스)
- ChromaDB (벡터 검색)
- 아이템 벡터 테이블

### 18.2 사전 준비

**로컬 개발 환경:**
- Docker (최신 버전 권장)
- NVIDIA GPU (로컬 테스트 시) + NVIDIA Driver + `nvidia-container-toolkit`

**RunPod 환경:**
- RunPod 계정 및 GPU Pod 생성 권한
- Docker Registry (Docker Hub 또는 GitHub Container Registry 권장)
- RunPod Volume (영속 스토리지, 운영 환경에서 필수)

### 18.3 프로젝트 구조

```
AI/
├── Dockerfile                      # 컨테이너 이미지 빌드 설정
├── bentofile.yaml                  # BentoML 서비스 정의
├── requirements.txt                # Python 의존성
├── service.py                      # BentoML 서비스 엔트리포인트
├── app/                            # 추천 로직 모듈
│   ├── models/
│   │   ├── __init__.py
│   │   ├── loader.py               # 데이터를 로드
│   │   ├── sasrec_twotower.py      # TwoTower 모델
│   │   └── README.md
│   └── utils/
│       ├── __init__.py
│       ├── chroma_store.py         # ChromaDB 저장 로직
│       ├── clip_embedder.py        # 작품 임베딩 벡터
│       ├── config.py               # 환경 설정
│       ├── item_vector_table.py    # 아이템 벡터 저장 테이블
│       ├── mapping_store.py        # idx <=> artwork 맵핑
│       ├── recommender.py          # 추천 시스템
│       └── schemas.py              # ChromaDB 데이터 형식
└── artifacts/                      # 런타임 필수 아티팩트
    ├── checkpoints/
    │   ├── BEST_SASRec_model.pth
    │   └── BEST_BestRecommend_model.pth
    ├── mappings/
    │   ├── piece_index.json
    │   └── idx_to_piece.json
    ├── chroma_db/                  # ChromaDB 영속 데이터
```

> **절대 금지사항:**
> - 환경 A의 매핑 파일 + 환경 B의 체크포인트/벡터 조합
> - 매핑 파일 없이 체크포인트만 교체
> - 운영 중 매핑 파일 임의 수정

```bash
# Volume 데이터 절대 덮어쓰지 않도록 주의
```

### 18.9 RunPod 문제 해결

#### Q1: 외부 접속 불가

**증상:** `curl: (7) Failed to connect`

**체크리스트:**
- RunPod에서 **Expose Ports = 8000** 설정했는가?
- 서버가 `0.0.0.0:8000`에 바인딩되는가? (로그 확인)
- Pod 로그에 `Listening on 0.0.0.0:8000` 메시지 있는가?
- RunPod Public URL이 정확한가?

#### Q2: 체크포인트 로드 실패

**증상:** `RuntimeError: Error(s) in loading state_dict`

**원인:**
1. 매핑 파일(`piece_index.json`)과 체크포인트의 `num_items` 불일치
2. `FF_DIM` 등 모델 하이퍼파라미터 변경
3. 서로 다른 환경의 아티팩트 혼용

**해결:**
```bash
# 매핑 파일 개수 확인
cat /workspace/artifacts/mappings/piece_index.json | jq 'length'

# 모델 로드 로그 확인
grep "num_items" /var/log/service.log
```

#### Q3: ChromaDB 벡터 검색 이상

**증상:** `chroma_count: 0` 또는 추천 결과 없음

**해결:**
```bash
# Chroma DB 디렉토리 확인
ls -la /workspace/artifacts/chroma_db/

# 권한 문제 해결
chmod -R 755 /workspace/artifacts/chroma_db/
```

### 18.10 Best Practices

#### Volume 기반 Artifacts 관리

RunPod Volume에 artifacts를 영구 보관하고, 컨테이너 이미지에는 포함하지 않습니다 (운영 시).

**이유:**
- 모델/매핑 업데이트 시 이미지 재빌드 불필요
- ChromaDB는 운영 중 지속적으로 변경됨
- 재배포 시에도 학습 상태 유지

#### 매핑 파일 버전 관리

```bash
# Git에서 매핑 파일 버전 관리
git add artifacts/mappings/
git commit -m "Update mappings: add artwork 1000-1050"

# 새 작품 추가 시 append-only
# 기존 인덱스 절대 변경 금지
```

#### 환경별 설정 분리

```bash
# .env.dev
DEVICE=cpu
BATCH_SIZE=16

# .env.prod (RunPod)
DEVICE=cuda:0
BATCH_SIZE=64
```
